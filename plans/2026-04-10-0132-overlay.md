# Overlay Decomposition Plan

**Last Updated:** 2026-04-10
**Status:** Decomposition Implemented, Automated QA Mostly Landed, Manual Windows QA Pending

## Summary

This plan decomposes the overlay feature into implementation-sized pieces that
match the current WhisperTray architecture. The overlay is a single optional
Qt-backed status surface. It is not a separate app mode and it does not have
multiple visual styles.

Implementation must follow [prompts/CODESTYLE.md](../prompts/CODESTYLE.md) and
the checked-in tool configuration.

## Current Progress

### Latest Checkpoint

- The overlay/runtime boundary now uses a typed `OverlaySettings` value object
  instead of loose `enabled` / `position` / `screen_target` arguments
- `WhisperTrayApp` now builds overlay runtime settings from one shared helper
  before recreating live controllers
- Both the legacy threaded overlay path and the shared Qt tray runtime now

- The decomposition below now matches the checked-in implementation
- Remaining work is limited to manual Windows QA and packaged-build validation

## Decomposition

### 1. Shared state and presentation layer

Purpose: keep all user-visible overlay behavior driven by the same typed state
model already used by the tray.

Scope:

- `AppState`, `AppStateSnapshot`, and `AppStatePresentation` remain the source
  of truth for overlay content.
- The overlay consumes presenter output only. It does not format business logic
  on its own.
- `TRANSCRIBED` remains a first-class state so the overlay can show recognized
  text until clipboard ownership changes.

Files:

- `whisper_tray/state.py`
- `whisper_tray/app.py`
- `whisper_tray/clipboard.py`

Progress:

- Implemented

### 2. Overlay controller and fallback boundary

Purpose: keep the optional UI dependency isolated from app orchestration.

Scope:

- `NullOverlayController` stays the safe no-op fallback when Qt is unavailable
  or overlay startup fails.
- `ThreadedOverlayController` remains the bridge for the legacy `pystray` path.
- Runtime creation is parameterized only by enablement, position, and display
  target.

Files:

- `whisper_tray/overlay/controller.py`

Progress:

- Implemented
- Runtime creation now flows through a typed `OverlaySettings` object so the
  fallback boundary is explicit and stable

### 3. Qt overlay window implementation

Purpose: own the actual on-screen rendering and placement behavior.

Scope:

- Keep one overlay window implementation: the anchored status card.
- Preserve the current overlay responsibilities:
  - render presenter output
  - auto-hide when requested
  - stay topmost and passive
  - follow the configured position and display target
  - recover sensibly across multi-monitor lookup failures
- Keep Windows-specific click-through and DPI helpers in this module.

Files:

- `whisper_tray/overlay/pyside_overlay.py`

Progress:

- Implemented

### 4. App and tray integration

Purpose: wire the optional overlay into the existing runtimes without spreading
UI decisions across the codebase.

Scope:

- The app layer owns enabling, disabling, and recreating the overlay when tray
  settings change.
- The tray menu owns only user controls:
  - enabled
  - position
  - display target
  - ready auto-hide
  - density
- The Qt tray runtime reuses one shared overlay host on the same event loop.
- `TRAY_BACKEND` continues to select Qt vs legacy `pystray` runtime behavior.

Files:

- `whisper_tray/app.py`
- `whisper_tray/tray/menu.py`
- `whisper_tray/tray/runtime.py`
- `whisper_tray/config.py`

Progress:

- Implemented

### 5. Verification and release gating

Purpose: make overlay work measurable instead of subjective.

Automated checks:

- state/presenter tests for overlay-facing copy
- overlay-controller fallback tests
- overlay geometry and multi-monitor fallback tests
- tray/runtime integration tests for overlay recreation and state fan-out

Manual Windows QA:

1. Verify with `TRAY_BACKEND=qt`
2. Verify with `TRAY_BACKEND=pystray`
3. In both cases confirm:
   - overlay stays topmost and click-through
   - overlay does not steal focus
   - hotkey flow records, transcribes, and pastes end to end
   - `TRANSCRIBED` persists until clipboard ownership changes
   - `OVERLAY_SCREEN=primary` and `OVERLAY_SCREEN=cursor` both behave correctly
   - packaged builds behave the same as source mode

Progress:

- Automated coverage for the overlay decomposition is in place
- Manual Windows QA remains open

## Out of Scope

- multiple overlay styles
- full-screen overlay redesign work
- new overlay states beyond the current shared state model
- moving config reads out of `config.py`
- replacing the fallback-oriented runtime split
